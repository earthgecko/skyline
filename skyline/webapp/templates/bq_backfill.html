BigQuery :: backfill

{% if bq_enabled|string == 'False' %}
BigQuery is not enabled on this Skyline node
{% endif %} A BigQuery backfill job is a one-off task that allows you to backfill BigQuery data into Skyline BEFORE enabling Vista to fetch the BigQuery data periodically.
To initiate a BigQuery backfill job you must first define the settings for the BigQuery data in settings.py VISTA_BQ_ACCOUNTS dict with the schedule values all set to 0, to ensure that source is not enabled in Vista. Enter the settings VISTA_BQ_ACCOUNTS key relating to the BigQuery data and the relevant timestamps which you want to backfill from and until.
This backfill function is based on the premise that each query will be made on a WHERE condition that is filtering on the DATE/TIMESTAMP field as each interval.
As such the Replace query string MUST contain the string DATE_FORMAT in place of the actual date or timestamp, int he backfill job this will be replaced for each interval with the DATE_FORMAT that is given.
For example if your settings had the query value set as:
SELECT * FROM mytable WHERE date = DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY)
with the DATE_FORMAT of %Y-%m-%d, from_timestamp of 1684191600, until_timestamp of 1684450800, interval of 86400 and replace_query_string of DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY)
The backfill job would run the following queries and submit the data for each query:
SELECT * FROM mytable WHERE date = '2023-05-16' # and would create metrics from the data and post metrics to flux
SELECT * FROM mytable WHERE date = '2023-05-17' # and would create metrics from the data and post metrics to flux
SELECT * FROM mytable WHERE date = '2023-05-18' # and would create metrics from the data and post metrics to flux
SELECT * FROM mytable WHERE date = '2023-05-19' # and would create metrics from the data and post metrics to flux
By default dry_run is set to true so that the default is to test not do. When a job is submitted with dry_run true the entire process is run but the data is only submitted to flux as test data, it is retrieved from BigQuery, converted into metric format and submitted to flux in test mode and returned in the job status_url page.
The backfill task is done in a sensible way and does not submit ALL the data this ensures that the system is not subjected to a thundering herd scenario.
The status_url will return the status of the task and report percent_complete and elapsed time.
Option value
VISTA_BQ_ACCOUNTS key The key of the settings to use from settings.VISTA_BQ_ACCOUNTS in settings [required]
from_timestamp the unix timestamp of the date and time to start the backfill from [required]
until_timestamp the unix timestamp of the date and time to end the backfill on (will be included as the last interval query)[required]
interval the interval at which to query the data e.g. daily would be 86400 [required]
DATE_FORMAT the date format to be used in the SQL at each interval selector replaced with date format to use, this must be a valid date string format e.g. %Y-%m-%d [required]
replace_query_string the string to be replaced in the query element of the account settings by DATE_FORMAT [required]
zero_fill NOTE this by default is set to false. Submit 0 when there is no value for a metric for an interval. Consider rather setting the zero_fill_analysis in the VISTA_BQ_ACCOUNTS settings for the job.
use_archive_data NOTE this by default is set to true so if archive data files exist for the period they will be used.
max_creates_per_minute the maximum number of new metrics to create per minute. This value must align with your Graphite MAX_CREATES_PER_MINUTE value defined in your Graphite's carbon.conf otherwise metrics will be dropped on Graphite. [required]
replace_redis_timeseries NOTE this by default is set to false. If you set this to true the entire Redis time series will be overwritten. This should only be used if you are replacing test data with a real backfill.
dry_run NOTE this by default is set to true
flux_test NOTE this by default is set to true, if true will be submitted to flux but in test mode only e.g. flux will accept the data and report what was accepted but will not process it.
verify_ssl NOTE this by default is set to true, if you are using a self-signed SSL set this to false.
